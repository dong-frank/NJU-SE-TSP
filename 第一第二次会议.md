会议主题​：大模型代码补全评测系统项目启动与分工研讨会
​发言人​：饶东申、张珂铫、邵勃源、刘汇鑫
​会议摘要​：本次会议明确了项目目标为开发大模型代码补全评测系统，制定了MVP功能范围与两周交付计划，确立了公平性、可视化及离线推理三大核心指标，并完成了TSP角色分工。

一、项目目标与MVP定义
针对“大语言模型代码补全能力测评系统”的开发需求，团队明确了产品定位及最小可行产品（MVP）的功能边界：

1. 核心产品定位与业务目标
​产品形态定义​：构建一个 Web 应用，专门用于评测大语言模型（LLM）的代码补全能力，区别于传统的 OJ 平台，重点在于模型能力的量化评估。
​业务价值主张​：解决学生作业提交与助教评分的痛点，提供统一的离线推理服务，确保评测过程的公平性与可复现性。
2. MVP 功能范围与交付标准
​核心功能清单​：MVP 需包含前端 Web 页面、统一后端推理服务、学生提交与分数查看、助教查看排行榜及作业详情、以及基于学号/ID 的搜索功能。
​交付时间节点​：设定两周内完成 MVP 开发，目标是交付一个可供助教试用的 Demo，要求系统具备严谨性，能有效防止作弊和攻击。
二、技术实现与评测指标
为确保评测系统的科学性与实用性，会议对后端架构及评测标准进行了详细规划：

1. 后端推理与架构规范
​离线推理模式​：采用离线推理架构，学生仅需上传模型权重和推理脚本，系统负责构建模型并运行，而非通过 API 调用。
​资源与安全限制​：为防止资源滥用，系统需限制模型大小及推理时间，并在提交前进行文件完整性校验，避免无效计算占用资源。
2. 评测指标与公平性设计
​多维度评测指标​：除代码相似度外，需引入时间开销（吞吐量）、Token 消耗、算力消耗等硬性指标，并考虑引入大模型辅助评估代码质量。
​防作弊机制​：严格保护测试集，防止学生通过过拟合或硬编码刷分；同时限制学生仅能查看分数和自身报告，禁止查看他人模型权重及详细代码。
三、TSP 角色分工与职责
依据 TSP（团队软件过程）模型，团队成员进行了角色分配，明确了各自在项目周期内的核心职责：

​质量经理​：由张珂铫担任，负责组织质量计划会议，重点关注评测指标的公平性、可复现性及代码评审机制。
2. 开发与支持角色分配
​开发经理 & 支持经理​：由邵勃源担任，负责制定开发策略（前后端技术选型）、划定 MVP 范围及环境配置。
​计划经理​：由刘汇鑫担任，负责组织计划会议、估算工作量、维护进度表及识别阻塞点。
