# 团队会议纪要

> 本文用于维护 TSP Launch 会议过程记录。它记录已经发生的会议事实、讨论、决策、未决问题和行动项；未召开的会议不提前生成结论。`项目策划方案.md` 只沉淀已经由会议确认的正式方案内容。

## 0. 维护规则

1. 每次会议结束后，先更新本文，再把已经确认的会议产物同步到 `项目策划方案.md`。
2. 本文只记录已经发生的会议，不为未召开会议补写 WBS、质量计划、风险清单或个人计划。
3. 如果原始记录把多个会议混在一起，应按 TSP 会议主题拆分为对应 Meeting。
4. 每次会议至少记录：会议主题、参会人、记录来源、会议目标、关键讨论、已形成决策、未决问题、行动项、可同步到策划方案的位置。
5. 阶段二答题只能引用阶段一已经沉淀到 `项目策划方案.md` 的内容；本文作为过程证据和追溯依据。

### 2.2 会议目标

明确项目要建设什么平台、服务什么课程场景、两周 MVP 应覆盖哪些能力，以及什么结果可以支撑后续策划书和阶段二答辩。

### 2.3 关键讨论

1. 团队确认项目形态是 Web 应用，专门用于评测大语言模型的代码补全能力。
2. 团队确认平台区别于传统 OJ，重点不是判断手写代码是否通过测试，而是量化模型的代码补全能力。
3. 团队讨论了学生提交与助教评分的痛点，提出通过统一离线推理服务提升公平性和可复现性。
4. 团队初步划定 MVP 功能范围，包括前端 Web 页面、统一后端推理服务、学生提交与分数查看、助教排行榜和作业详情、学号/ID 搜索。
5. 团队提出评测应关注公平性、可视化、离线推理，并关注作弊防护和攻击防护。
6. 团队讨论后端推理与评测指标，认为应限制模型大小和推理时间，并在提交前进行文件完整性校验。
7. 团队认为评测指标不应只看代码相似度，还应考虑时间开销、Token 消耗、算力消耗等硬性指标。

### 2.4 已形成决策

| 决策项 | 会议结论 |
| --- | --- |
| 产品形态 | 构建 Web 应用，用于评测 LLM 代码补全能力 |
| 产品定位 | 区别于传统 OJ 平台，重点是模型能力量化评估 |
| 业务价值 | 解决学生提交与助教评分痛点，提供统一离线推理服务 |
| MVP 周期 | 两周内完成 MVP，目标是交付可供助教试用的 Demo |
| MVP 功能 | 前端 Web 页面、统一后端推理服务、学生提交与分数查看、助教排行榜和作业详情、学号/ID 搜索 |
| 核心质量方向 | 公平性、可视化、离线推理、防作弊、防攻击、可复现 |
| 推理方式 | 采用离线推理，学生上传模型权重和推理脚本，由系统构建并运行 |
| 资源限制 | 限制模型大小和推理时间，提交前进行文件完整性校验 |
| 评测指标 | 除代码相似度外，引入时间开销、Token 消耗、算力消耗等指标 |

### Meeting 5：质量计划

#### 4.1 基本信息

| 项目 | 内容 |
| --- | --- |

| 主持人 | 张珂铫，质量经理 |

| 会议时间 | 6.10 21:15~21:35 |
| 对应 TSP 会议 | Meeting 5：质量计划 |
| 记录完整度 | 较高。已包含质量目标、评审计划、缺陷管理、质量预警和会后行动项；具体评审时间点和每项评审责任人仍需在 Meeting 6 计划平衡中进一步落地 |

#### 4.2 会议目标

建立 TSP 风格质量计划，明确项目策划方案和 MVP 的质量目标，确定评审对象、缺陷分级、缺陷记录规则、质量预警机制，以及需要带入 Meeting 6 的质量资源和计划平衡事项。

#### 4.3 关键讨论

1. 张珂铫强调第五次会议不只是安排最后测试，而是要围绕两周 MVP 计划建立质量保障体系。
2. 团队确认质量目标分为两类：一类是项目策划方案质量，要求完整覆盖目标、策略、风险和承诺；另一类是 MVP 质量，重点关注模型校验过程、主链路复现和评测结果可信度。
3. 团队确认助教评测闭环不能只以“功能完成”为验收标准，还需要检查接口规范、数据格式、版本环境、指标口径和榜单安全。
4. 张珂铫提出在开发前通过评审提前发现问题，评审重点包括项目策划方案完整性、MVP 范围、每日计划、主链路技术细节、评测指标和排行榜指标一致性。
5. 团队确认 Day 12-14 用于整体联调、缺陷修复、部署和验收，质量活动不能被新增功能挤占。
6. 团队讨论缺陷管理方式，确认采用 P0、P1、P2 三个等级，并要求缺陷记录包含问题描述、位置、严重级别、责任人、复现方式和修复状态。
7. 张珂铫提出质量预警机制，饶东申建议将缺陷评审流程嵌入每日站会，形成即时质量反馈。
8. 团队讨论功能自测和集中评审之间的平衡，确认开发同学完成功能后需要记录测试情况和截图，质量经理负责识别虚假测试或缺少证据的情况。
9. 团队确认每日代码合入前需要通过质量经理审核，质量经理在日常站会上持续把控质量情况。

#### 4.4 已形成决策

| 决策项 | 会议结论 |
| --- | --- |
| 质量原则 | 质量前移，通过评审、过程检查、缺陷记录和质量预警保障质量，而不是依赖最后测试兜底 |
| 策划方案质量目标 | 策划方案应完整覆盖项目目标、开发策略、质量计划、风险处理和团队承诺，并能支撑阶段二答题 |
| MVP 质量目标 | MVP 重点保证模型校验清晰可信、主链路可复现、指标口径一致、榜单安全、任务状态和失败原因可追踪 |
| 评审对象 | 项目策划方案、MVP 范围、每日计划、主链路环境版本、评测指标与排行榜指标口径、接口规范和数据格式 |
| 缺陷分级 | P0 为严重阻塞 MVP 实现；P1 为影响核心质量；P2 为轻微问题 |
| 缺陷记录要求 | 缺陷记录需包含问题描述、位置、严重级别、责任人、复现方式和修复状态 |
| 质量预警机制 | 策划方案问题、联调修复时间不足、主链路接入不确定、指标导致排行榜异常、数据泄露风险、P0/P1 缺陷未及时解决时，由质量经理向项目组长预警 |
| 日常质量把控 | 缺陷评审嵌入每日站会，功能完成后记录测试情况和截图，代码合入需通过质量经理审核 |
