# FIM 评测平台需求沉淀

> 来源：`FIM评测平台需求/REQ-001-fim-platform-requirements-release.md` 及其配套图示。
>
> 用途：为本课程作业的 TSP 启动会议、《项目策划方案》和阶段二答题提供需求依据。本文不是原需求文档的逐字复制，而是面向策划书写作的结构化提炼。

## 1. 一句话需求理解

本项目要建设一个面向课程打榜场景的 FIM（Fill-in-the-Middle）代码补全能力测评平台，把学生模型权重提交、模型目录校验、推理后端选择、批量评测、指标计算、报告聚合和公开榜单发布组织成一套可审计、可复现、可扩展、可部署的服务化流程。

策划书中应避免把它写成普通“模型评测脚本”。它的本质是：

- 面向学生：提交模型，查看本人结果和公开榜单。
- 面向助教：批量拉取提交，统一评测，追踪状态，发布榜单。
- 面向管理员：维护部署、权限、Secret、后端适配和故障恢复。

## 2. 背景与痛点

FIM 代码补全比一次性生成完整程序更接近真实 IDE 场景。模型需要同时理解代码前缀和后缀，在有限上下文和低延迟约束下补出中间片段。

需求文档强调，FIM 评测不能只看单一正确率。一个可用于真实代码补全场景的模型，还需要关注：

- 是否复读前缀或后缀。
- 是否迟迟不停止，导致 TLE。
- 平均生成 token 数是否可控。
- 多测试集和多任务口径下指标是否可复现。
- 评测环境、提交遗漏、异常状态和榜单口径是否可追踪。

当前痛点是：如果仍依赖助教手工下载权重、检查目录、运行不同后端、拼接报告，容易出现环境漂移、遗漏提交、异常不可追踪、榜单口径不一致等问题。

因此，本平台的价值主张是：把已有评测能力服务化、流程化、审计化，而不是从零重写评测器。

## 3. 系统目标

### 3.1 产品目标

- 支持学生按课程要求提交打榜模型权重。
- 支持助教按课程、作业、学生名单和日期策略批量拉取提交。
- 支持模型根目录校验，提前识别上传父目录、缺配置、缺 tokenizer、缺权重等错误。
- 支持多个 FIM/代码补全测试集和任务口径。
- 支持多推理后端，通过统一契约接入 MindSpeed、vLLM、OpenAI-compatible completions、diffusion / fast_dllm 等。
- 支持 generation JSONL 恢复、指标计算、复读/TLE 分析。
- 支持内部聚合报告、可视化和公开榜单发布。
- 支持状态管理、审计复现、实时日志和后续服务化接口。

### 3.2 业务目标

- 让助教能按统一规则处理整班提交。
- 让学生能看到公开、稳定且不泄露内部数据的评测结果。
- 让榜单结果可复现、可解释、可审计。
- 支持两周内优先交付可用 MVP，后续再扩展为完整在线服务。

## 4. 系统边界与外部依赖

### 4.1 系统边界

FIM 测评平台内部包含：

- Web 控制台 / API。
- `ta-client`：以助教身份申请只读 STS，并从 OBS 拉取 submissions。
- 模型根目录校验器。
- 任务编排器：队列、幂等、状态机。
- 评测 Worker：prompt 导出、调用后端、指标计算。
- 后端适配器：统一生成结果格式。
- 报告与榜单发布器。

### 4.2 外部依赖

- `issuer-server`：外部 STS 签发服务。学生和助教都通过临时 STS 访问 OBS。
- 华为云 OBS：保存学生 submissions、评测 artifacts、公开 leaderboard。
- 华为云 SWR：保存课程镜像或学生自建镜像。
- 推理后端与运行环境：MindSpeed / vLLM / OpenAI-compatible / diffusion / fast_dllm。
- 华为云 ModelArts / Notebook / NPU 资源：作为训练、评测和部署环境。

### 4.3 权限边界

- 学生申请上传 STS 后上传模型权重到 OBS。
- 平台内 `ta-client` 以助教身份申请只读 STS，用于列举和下载 submissions。
- 长期 AK/SK 不应进入平台普通运行环境。
- 公开榜单不应泄露本地绝对路径、模型路径、tokenizer 路径、逐样本内容和内部日志。

## 5. 核心业务流程

### 5.1 学生训练与提交

学生流程可概括为：

1. 登录轻量算力节点。
2. 使用课程现成镜像，或自建 Docker 镜像。
3. 下载基础权重。
4. 在容器内完成模型后训练。
5. 在容器内进行本地评估和闭环调优。
6. 将最终打榜模型权重上传到华为云 OBS。

对平台的要求是：理解容器化训练产物、OBS 提交路径、模型根目录结构和最终打榜权重之间的关系。

### 5.2 助教评测闭环

助教流程可概括为：

1. 创建同步任务，指定课程、作业、名单和日期策略。
2. 平台通过 `ta-client` 获取助教只读 STS。
3. 从 OBS 拉取学生 submissions。
4. 校验模型根目录。
5. 创建评测任务，选择任务集、推理后端、样本限制和评分策略。
6. 导出 prompt，调用推理后端。
7. 回收 generation JSONL、progress JSON 和日志。
8. 恢复结果并计算指标。
9. 生成内部报告、失败归因和可视化。
10. 发布过滤敏感信息后的公开榜单。

### 5.3 管理员运维流程

管理员负责：

- 平台部署、镜像、配置 Profile 和 Secret 引用。
- 推理后端适配和资源池维护。
- 权限与公开榜单策略配置。
- 故障处理、任务恢复、日志追踪和榜单回滚。

## 6. 推理后端契约

平台不要求所有推理后端采用同一种模型加载方式、并行策略或服务形态。它只要求后端接收统一输入并返回统一输出。

### 6.1 统一输入

- `prompt_file`：标准 prompt JSONL。
- `model_ref`：本地权重目录、MCore 目录或远程模型标识。
- `backend`：MindSpeed、vLLM、OpenAI-compatible completions、diffusion、fast_dllm 等。
- `sampling`：temperature、top_p、max_tokens、stop 等采样参数。
- `runtime`：batch size、设备、并行参数、超时、日志路径等。

### 6.2 统一输出

- `generations_file`：每条样本对应 prompt、生成文本、状态和可选 TLE 信息。
- `progress_file`：样本进度、批次进度和 ETA。
- `logs`：生成日志与评测日志。
- `metadata`：后端版本、模型来源、运行参数和失败原因。

### 6.3 对策划书的意义

两周 MVP 不宜承诺完整支持所有后端。更合理的策略是：

- 以 MindSpeed 作为推荐主链路。
- 保留统一后端契约和适配器边界。
- 用 mock backend 或已有 raw result 支持演示与容错。
- 把 vLLM、OpenAI-compatible、diffusion / fast_dllm 作为可扩展路线。

## 7. 功能需求提炼

| 编号 | 功能模块 | 策划书表达重点 |
| --- | --- | --- |
| FR-01 | 课程提交拉取 | 通过 `ta-client` 获取助教只读 STS，按课程、作业、名单、日期同步提交。 |
| FR-02 | 模型根目录校验 | 检查 `config.json`、`tokenizer.json`、权重文件等，提前发现目录层级错误。 |
| FR-03 | 多测试集评测 | 支持配置化选择 FIM/代码补全测试集、评分模式和样本范围。 |
| FR-04 | 多推理后端 | 用统一契约封装后端差异，MVP 可优先支持主链路。 |
| FR-05 | 结果恢复与指标计算 | 将 generation JSONL 转为 evaluator 可读输入，计算核心指标。 |
| FR-06 | 内部聚合报告 | 输出 JSON/CSV/Markdown 报告、可视化网页和基线对比。 |
| FR-07 | 公开榜单 | 生成安全过滤后的公开 HTML 榜单。 |
| FR-08 | 状态管理 | 标记 ok、absent、tokenizer_error、vocab_error、missing_raw_result 等状态。 |
| FR-09 | 审计与复现 | 保存配置、后端、任务、样本数、日志、指标和评分源码。 |
| FR-10 | 服务化接口规划 | 规划同步、评测创建、进度查询、榜单查询、artifact 下载接口。 |
| FR-11 | 实时日志可视化 | 助教实时查看阶段、日志、进度、失败原因；学生只看授权摘要。 |

## 8. 非功能需求提炼

| 类别 | 关键要求 | 策划书可用表述 |
| --- | --- | --- |
| 安全性 | 长期 AK/SK 不进入平台普通运行环境，只使用临时凭证 | 采用最小权限和 SecretRef 管理 |
| 隔离性 | 内部日志、逐样本结果、公开榜单分层 | 公开页面过滤敏感路径和样本内容 |
| 可靠性 | STS 过期、OBS 中断、Worker 崩溃、后端超时要可重试或明确失败 | 任务状态机、幂等和失败归因 |
| 可用性 | 后端不可用时仍能演示报告和榜单链路 | mock backend、sample result、已有 raw 结果 |
| 可扩展性 | 新增后端不改 evaluator 主体 | 后端注册和统一适配器 |
| 高效性 | 增量下载、分批执行、MCore 权重复用、batch size 和并发控制 | 降低重复计算和资源浪费 |
| 可观测性 | 各阶段有进度、日志、trace id、失败原因 | 浏览器实时回传和任务追踪 |
| 可维护性 | 推荐链路、历史链路、配置和镜像边界清晰 | 避免依赖混装和硬编码 |
| 可复现性 | 保存数据集、评分模块、后端版本、运行参数和配置摘要 | 支持重算和审计 |
| 兼容性 | 记录 CANN、torch_npu、MindSpeed、vLLM、OpenAI SDK 等版本 | 支持回溯和隔离 |
| 公平性 | 同一榜单批次固定数据集、评分模块、后端参数和截止策略 | 保证排名口径一致 |
| 可部署性 | API、Worker、前端、榜单服务、后端容器化 | 支持本地、助教 Notebook、后续集群化 |

## 9. 角色与用例

### 9.1 学生

主要用例：

- 提交模型权重。
- 查看本人评测结果。
- 查看公开榜单。

学生侧权限应受限：只能查看本人授权结果和已发布公开榜单，不能访问他人结果、内部日志、逐样本内容或隐藏测试集。

### 9.2 助教

主要用例：

- 拉取与校验学生提交。
- 创建并运行评测任务。
- 查看实时日志。
- 生成内部报告。
- 发布公开榜单。

助教是平台最关键用户。两周 MVP 应优先保证助教评测闭环可用。

### 9.3 管理员

主要用例：

- 管理系统部署。
- 维护后端适配。
- 配置权限与 Secret。
- 处理故障与恢复。

管理员用例多与非功能需求相关，是策划书中风险、支持计划和部署计划的重要依据。

## 10. 指标与排名规则

平台测量学生提交模型在测试集上的 FIM 能力，并区分排名指标与审计指标。

| 指标 | 含义 | 默认公开 Score 中的作用 |
| --- | --- | --- |
| EM / exact_match_rate | 生成补全与参考答案完全一致比例，越高越好 | 权重 0.20 |
| ES / average_edit_similarity | 生成补全与参考答案编辑相似度，越高越好 | 权重 0.70 |
| ES_repoeval | RepoEval 风格编辑相似度 | 内部分析，不直接参与默认公开 Score |
| TLE 通过率 | 由 TLE 比例折算，越高越好 | 权重 0.05 |
| 非复读率 | 由前缀/后缀复读比例折算，越高越好 | 权重 0.05 |
| total、生成 token 数、任务状态 | 样本量、生成长度和状态审计 | 不直接加分 |

默认公开榜单综合分：

```text
Score = 0.20 * EM + 0.70 * ES + 0.05 * TLE通过率 + 0.05 * 非复读率
```

排名规则：

- 只有 `status=ok` 且 EM、ES 可计算的提交参与排名。
- TLE 通过率或非复读率缺失时，不直接按 0 分处理，而是按可用权重归一。
- 排名先按 Score 降序，再按 ES 降序，仍相同则按 student_id 升序。

对策划书的意义：质量计划不能只写“测试通过”，还要保证指标口径、评分模块、后端参数、样本范围和榜单规则固定可审计。

## 11. 现有实现与目标差距

需求文档说明系统已有不少能力，但目标是服务化和流程化。

| 领域 | 已有能力 | 主要差距 | 策划书启示 |
| --- | --- | --- | --- |
| MindSpeed 评测链路 | 脚本、桥接、HF 到 MCore 转换、批量生成、结果恢复、指标计算 | 尚非 API + 队列 + Worker 在线系统 | MVP 可复用脚本，计划中补服务化封装 |
| 多测试集 | 已有多个任务入口 | 未纳入统一配置中心和白名单 | 先做配置化选择，再扩展管理 |
| 多后端 | MindSpeed 推荐，其他后端有兼容参考 | 缺统一注册、契约校验、资源调度和容器隔离 | MVP 先主链路，保留后端适配边界 |
| 指标与重算 | 可计算 EM/ES/复读/TLE，可从 raw 重算 | 元数据和评分源码快照未强制固化 | 质量计划中加入 artifact 元数据要求 |
| 批量学生模型 | 可扫描 submissions 并生成脚本、报告 | 未直接集成 `course-obs-sts` 的 `ta-client` | 提交同步是关键风险和核心任务 |
| 报告与榜单 | 可生成可视化、聚合报告、公开榜单 | 缺 Web 控制台、鉴权查分、实时日志、下载 API、回滚服务 | 两周计划需裁剪功能并强调分期 |
| 校验与测试 | 有部分依赖检查和模型校验测试 | 缺服务化端到端、权限、队列恢复、资源配额、故障演练 | 质量计划应覆盖服务化后的关键测试 |

## 12. 两周 MVP 范围建议

### 12.1 建议纳入 MVP

两周内建议优先覆盖助教闭环：

- 学生 submissions 同步任务。
- 模型根目录校验。
- 评测任务创建和状态管理。
- MindSpeed 主链路或 mock/raw 演示链路。
- generation JSONL 恢复和核心指标计算。
- 内部聚合报告。
- 公开榜单生成。
- 基础实时日志或阶段进度展示。
- 配置和运行元数据保存。

### 12.2 建议延期或降级

- 多后端完整资源调度。
- 学生完整鉴权查分门户。
- 复杂 artifact 下载 API。
- 发布回滚服务的完整工程化。
- 集群化水平扩展。
- 全量端到端故障演练。

这些内容不应在策划书中消失，而应作为“后续迭代”或“风险缓解后的扩展目标”出现。

## 15. 策划书写作注意事项

- 不要把需求写成“开发一个榜单网页”这么窄。核心链路是提交、校验、评测、指标、报告、榜单。
- 不要把多后端完整实现作为两周内强承诺。更稳妥的表述是主链路优先，契约预留扩展。
- 不要忽略权限和敏感信息过滤。公开榜单安全是专家会关心的质量点。
- 不要只写功能需求。非功能需求中的可复现、公平性、可观测性、安全性，对阶段二答辩非常有用。
- 不要把“已有实现”当成完全可交付系统。策划书应体现复用已有脚本能力，同时补上服务化、状态管理、日志、权限、发布等差距。
- 不要让技术细节盖过 TSP。技术内容应服务于计划、估算、质量和风险论证。
