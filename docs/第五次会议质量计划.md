# 第五次会议质量计划

>
> 角色定位：本文由质量经理张珂铫整理，用于沉淀 Meeting 5 的正式质量计划，并作为后续更新 `团队会议纪要.md` 和 `项目策划方案.md` 的依据。

## 1. 会议定位

第五次会议对应 TSP Launch 的质量计划会议。会议目标不是重新讨论开发策略，也不是只安排最后测试，而是在第三、四次会议已经形成的 MVP 开发策略和两周计划基础上，建立质量保障体系。

本次会议确认：项目质量应通过评审、过程检查、缺陷记录和质量预警前移控制，而不是依赖最后测试兜底。

## 2. 质量目标

团队将质量目标分为两类。

| 类别 | 质量目标 |
| --- | --- |
| 策划方案质量 | 阶段一项目策划方案需要完整覆盖项目目标、开发策略、计划安排、质量保障、风险处理和团队承诺，并能够支撑阶段二答题 |
| MVP 质量 | 两周 MVP 需要保证助教评测闭环可信，重点关注模型校验清晰性、主链路可复现性、评测指标一致性、排行榜安全性、任务状态和失败原因可追踪性 |

## 3. 质量评审计划

团队确认以下对象必须进入质量评审：

| 评审对象 | 评审重点 |
| --- | --- |
| 项目策划方案 | 是否完整覆盖目标、策略、风险和承诺，是否能够作为阶段二答题依据 |
| MVP 范围 | 是否符合 4 人团队、2 周周期约束，是否存在过度承诺 |
| 每日开发计划 | 是否为联调、缺陷修复、部署和验收预留时间 |
| 主链路环境和版本 | Mindspeed 主链路的环境、版本、接口、数据格式和失败原因是否明确 |
| 评测指标和排行榜 | 评测指标与排行榜指标口径是否一致，结果是否可解释 |
| 接口和数据格式 | 服务化封装、模型校验、状态管理和日志追踪的数据契约是否清楚 |
| 榜单安全 | 公开榜单是否避免泄露模型路径、隐藏样本、内部日志或他人信息 |

## 4. 缺陷管理规则

团队采用 P0、P1、P2 三级缺陷管理。

| 级别 | 含义 | 处理要求 |
| --- | --- | --- |
| P0 | 严重阻塞 MVP 实现或影响核心交付 | 必须优先处理，未解决时应触发质量预警 |
| P1 | 影响核心质量、评测可信度或方案可信度 | 应及时处理，避免进入最终提交或演示 |
| P2 | 轻微问题，例如局部表述、格式或非核心体验问题 | 可排入后续修复，但需记录状态 |

缺陷记录至少包含以下字段：

| 字段 | 说明 |
| --- | --- |
| 问题描述 | 简要说明问题是什么 |
| 位置 | 对应文档位置、功能模块或页面位置 |
| 严重级别 | P0 / P1 / P2 |
| 责任人 | 负责修复或解释该问题的成员 |
| 复现方式 | 如何观察或复现该问题 |
| 修复状态 | open / fixed / verified / deferred |

## 5. 质量预警机制

质量经理在以下情况出现时，需要及时向项目组长预警：

1. 项目策划方案存在缺失、错误或前后不一致，影响阶段二答题。
2. 两周计划没有为联调、缺陷修复、部署和验收预留足够时间。
3. Mindspeed 主链路接入存在不确定性，且缺少备用演示方案。
4. 评测指标口径不清，导致排行榜结果异常或无法解释。
5. 公开榜单存在数据泄露风险，例如泄露模型路径、隐藏样本、内部日志或他人信息。
6. P0 或 P1 缺陷未及时解决，影响 MVP 可信度或最终交付。

预警后，质量经理应与项目组长、计划经理共同判断是否需要调整计划、压缩非核心范围，或将问题带入第六次会议进行计划平衡。

## 6. 日常质量把控

会议确认质量控制需要进入日常协作流程：

1. 每日站会中加入质量检查和缺陷评审。
2. 质量经理负责持续把控质量情况，关注开发进度和质量是否偏离计划。
3. 各开发同学完成功能后，需要记录测试情况和截图，证明功能完成且未影响已有功能。
4. 质量经理需要识别虚假测试或缺少测试证据的情况。
5. 每日代码合入前需要通过质量经理审核。
6. 团队成员需要熟悉主链路逻辑，保证功能自测不是孤立验证。

## 7. 带入第六次会议的事项

以下事项需要带入 Meeting 6 的个人计划和计划平衡：

| 事项 | 说明 |
| --- | --- |
| 质量活动工时 | 评审、联调、缺陷修复、部署和验收必须进入每日计划 |
| 责任人绑定 | 每项评审和缺陷处理需要明确责任人 |
| Day 12-14 保护 | Day 12-14 的整体联调和修复时间不能被新增功能挤占 |
| 主链路备用方案 | 如果 Mindspeed 主链路不稳定，需要准备备用演示方案 |
| P0/P1 处理机制 | P0/P1 缺陷未关闭时，需要判断是否调整范围或优先级 |

## 9. 可同步到策划方案的摘要

第五次会议由质量经理主持，目标是建立团队质量计划。团队确认，本项目质量保障不能依赖最后测试兜底，而应通过评审、过程检查、缺陷记录和质量预警前移控制。

团队将质量目标分为策划方案质量和 MVP 质量两类。策划方案质量要求阶段一方案完整覆盖项目目标、开发策略、质量保障、风险处理和团队承诺，并能支撑阶段二答题；MVP 质量要求助教评测闭环在模型校验、主链路复现、指标一致性、榜单安全和状态追踪方面可信。

团队决定对项目策划方案、MVP 范围、每日开发计划、Mindspeed 主链路、评测指标与排行榜、接口和数据格式、公开榜单安全进行评审。缺陷按 P0、P1、P2 管理，缺陷记录包含问题描述、位置、严重级别、责任人、复现方式和修复状态。

当出现策划方案问题、联调修复时间不足、主链路接入不确定、排行榜结果异常、数据泄露风险或 P0/P1 缺陷未及时解决时，质量经理向项目组长预警，并将相关问题带入第六次会议进行计划平衡。
