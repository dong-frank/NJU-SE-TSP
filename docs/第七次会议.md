# FIM评测平台项目第七次会议纪要

**会议主题**：风险评估与策划方案收口  
**会议方式**：团队讨论 + 会议材料整理  
**参会人员**：饶东申、张珂铫、刘汇鑫、邵勃源  
**主持/整理人**：饶东申（项目组长、过程经理）  
**对应TSP Launch会议**：Meeting 7 - 风险评估  

---

## 1. 会议定位

第七次会议是本次 TSP Launch 的风险评估会议。前六次会议已经分别完成产品目标、角色分工、开发策略、整体计划、质量计划、个人计划和计划平衡，本次会议不再新增功能范围，而是围绕已有目标和计划进行 “What if” 风险检查。

本次会议确认：风险评估的目的不是否定前期计划，而是提前识别可能导致项目策划方案失真、MVP范围失控、评测结果不可信或交付材料无法按时完成的因素，并给出触发条件、应对措施和责任角色。

---

## 2. 会议目标

本次会议主要完成以下目标：

1. 识别 FIM 评测平台项目在需求、时间、技术、质量、协作和交付方面的主要风险。
2. 明确每类风险的触发条件、影响程度、缓解措施和应急措施。
3. 将风险评估结果同步到《项目策划方案》，作为阶段二答题依据。
4. 明确策划方案最终补充内容和交付方式，保证 2026-06-12 前完成提交。
5. 确认七次会议记录如何作为策划方案的证据材料。

---

## 3. 风险评估方法

团队采用 TSP 第七次会议的 “What if” 思路，从关键假设出发评估风险：

| 评估问题 | 说明 |
| --- | --- |
| What if 需求理解有偏差？ | 如果仅基于文档理解需求，是否会误判平台边界或用户目标？ |
| What if 两周时间不够？ | 如果 MVP 范围过大，哪些内容必须裁剪？ |
| What if 主评测链路跑不起来？ | 如果推理后端、模型权重或环境不可用，是否有兜底方案？ |
| What if 指标或榜单不可信？ | 如果指标口径不清或计算错误，如何避免影响公平性？ |
| What if 团队任务不均衡？ | 如果关键成员过载，如何重新分配任务或调整范围？ |
| What if 策划书无法支撑阶段二？ | 如果阶段一没有预埋足够证据，阶段二如何避免新增内容？ |

风险等级采用“概率 / 影响”两维判断，分为高、中、低。责任角色不是唯一执行人，而是负责跟踪、预警和推动应对的人。

## 5. 重点风险处理策略

### 5.1 范围裁剪策略

会议确认，如果两周内资源不足，裁剪顺序应遵循“保核心闭环、降级展示细节、延期扩展能力”的原则：

1. 必须保留提交导入/同步、模型目录校验、任务状态、主推理链路或兜底链路、结果恢复、核心指标、内部报告和公开榜单。
2. 可以降级 OBS/STS 全自动同步，改为保留接口边界 + 样例数据或手动导入。
3. 可以降级实时日志可视化，先提供阶段进度、日志摘要和失败原因。
4. 可以降级学生侧能力，保留本人结果摘要和公开榜单，不做完整学生门户。
5. 可以延期完整多后端调度、集群化扩展、复杂 artifact 下载、完整榜单回滚和全量故障演练。

### 5.2 技术兜底策略

对于主推理链路风险，团队确认：

- FR-04 主链路是关键路径，但不应成为唯一交付条件。
- 若 MindSpeed 主链路或真实算力环境不可用，应启用 mock backend、sample results 或已有 raw 结果。
- 兜底链路的目的不是替代真实评测能力，而是保证“结果恢复、指标计算、内部报告、公开榜单”的后续链路可以演示和验证。

### 5.3 策划方案收口策略

会议确认，阶段一项目策划方案必须体现七次会议的过程和产物，否则无法证明团队是按 TSP 启动会议方式完成策划。

策划方案需要重点补充：

- 七次会议的组织人、主题和产出。
- MVP 将实现的基本功能。
- 现有实现与目标需求之间的差距。
- 2 周 MVP 的范围裁剪依据。
- 风险清单、触发条件、缓解措施和责任角色。
- 阶段二答题证据索引。

## 8. 可同步到策划方案的摘要

第七次会议由项目组长组织，全体成员围绕 TSP Launch 的 “What if” 风险评估方法，对 FIM 评测平台项目进行了集中风险识别。团队确认，本项目的主要风险包括需求理解偏差、两周 MVP 范围过大、主推理链路不稳定、学生提交异常、指标口径不清、榜单泄露敏感信息、团队任务不均衡以及策划方案无法支撑阶段二答题。

针对范围和时间风险，团队决定坚持助教评测闭环优先，将提交导入、模型目录校验、任务状态、主推理链路或兜底链路、结果恢复、核心指标、内部报告和公开榜单作为必须保留内容；将完整多后端调度、复杂学生门户、实时日志细节、复杂 artifact 下载、集群化扩展和完整回滚服务作为可降级或后续迭代内容。

针对技术实现风险，团队决定保留统一后端契约和 MindSpeed 主链路优先策略，同时准备 mock backend、sample results 或已有 raw 结果作为兜底方案，保证即使真实推理环境受阻，也能演示结果恢复、指标计算、报告和榜单链路。

针对质量与公平风险，团队决定固定测试集、评分模块、后端参数和榜单排序规则，并在公开榜单发布前执行敏感字段过滤，避免泄露模型路径、逐样本内容、内部日志或他人信息。质量经理负责指标口径和榜单安全预警。

会议最后确认，阶段一《项目策划方案》必须体现七次会议的组织过程、MVP 基本功能、现有实现与需求差距、风险清单和阶段二答题依据。会议纪要可作为附录或支撑材料，正式策划书则需要整合各次会议的关键产物，形成可提交、可答辩、可追踪的项目方案。
