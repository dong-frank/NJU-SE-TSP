## 1. 项目背景与价值主张

本项目面向课程打榜场景，拟建设一个用于评测大语言模型代码补全能力的 Web 应用。它不同于传统 OJ 平台，关注的不是学生手写程序是否通过测试，而是学生提交的大模型在 FIM（Fill-in-the-Middle）代码补全任务上的能力表现。

当前课程评测如果依赖助教手工下载模型权重、检查目录、运行评测脚本、拼接报告和发布榜单，容易出现环境漂移、提交遗漏、异常状态不可追踪、榜单口径不一致等问题。因此，本平台的核心价值是把已有评测能力服务化、流程化和审计化，使评测过程公平、可复现、可比较，并便于助教在统一规则下完成整班评测。

本项目策划的重点不是立即完成完整平台开发，而是按照 TSP（Team Software Process）启动会议方法，证明 4 人团队能够形成清晰目标、合理角色、可执行计划、质量策略、风险应对和共同承诺。

## 3. Meeting 1：产品目标与业务目标

### 3.1 会议目标

第一次会议目标是明确“要做什么”和“做到什么程度”。团队需要把模糊的“大语言模型代码补全能力测评系统”转化为可规划、可验收、可答辩的产品目标和业务目标。

### 3.2 已形成决策

| 决策项 | 会议结论 |
| --- | --- |
| 产品形态 | 构建一个 Web 应用，专门用于评测 LLM 代码补全能力 |
| 平台定位 | 区别于传统 OJ 平台，重点是模型能力量化评估 |
| 核心价值 | 解决学生提交与助教评分痛点，提供统一离线推理服务 |
| MVP 周期 | 2 周内完成 MVP，形成可供助教试用的 Demo |
| 核心指标 | 公平性、可视化、离线推理 |
| 质量要求 | 系统应严谨，能防止作弊和攻击，评测过程可复现 |

### 3.3 产品目标

本平台的产品目标是支持课程场景下的模型代码补全评测闭环：

1. 学生能够按课程要求提交模型权重和必要推理材料。
2. 系统能够进行模型目录与文件完整性校验，提前发现无效提交。
3. 系统能够通过统一后端推理服务执行 FIM 代码补全评测。
4. 系统能够计算代码补全能力相关指标，包括代码相似度、时间开销、Token 消耗、算力消耗等。
5. 学生能够查看本人成绩和报告摘要。
6. 助教能够查看排行榜、作业详情和学生提交状态。
7. 助教能够基于学号或 ID 搜索学生记录。
8. 平台能够保护测试集与学生模型，防止学生查看他人模型权重、详细代码或隐藏测试内容。

### 3.4 业务目标

本平台的业务目标是服务课程评测和专家验收：

1. 在 4 人团队和 2 周启动周期下，优先交付助教可试用的 MVP。
2. 将手工评测流程转化为统一、可追踪、可复现的服务化流程。
3. 让榜单结果在同一数据集、同一评分口径、同一评测参数下可比较。
4. 让学生只能看到授权范围内的分数和报告，保障公平性和安全性。
5. 让助教能够识别提交异常、评测失败和资源超时等情况，减少人工排查成本。
6. 为阶段二答辩提供可引用的目标、范围、计划、质量和风险依据。

### 3.5 MVP 范围

| 范围类别 | 纳入 2 周 MVP | 说明 |
| --- | --- | --- |
| 前端 Web 页面 | 是 | 支持学生查看分数、助教查看榜单和作业详情 |
| 后端推理服务 | 是 | 统一离线推理入口，优先保证主链路或演示链路可用 |
| 学生提交处理 | 是 | 支持提交记录接入、目录校验和状态展示 |
| 助教排行榜 | 是 | 展示公开榜单和基础筛选/搜索能力 |
| 作业详情 | 是 | 支持助教查看任务状态、学生结果和失败原因摘要 |
| 学号/ID 搜索 | 是 | 支撑助教快速定位学生提交 |
| 多维评测指标 | 是 | 至少覆盖核心能力指标和资源/时间类审计指标 |
| 完整多后端调度 | 否 | 作为后续扩展目标，MVP 保留统一契约边界 |
| 完整学生鉴权门户 | 否 | MVP 可先实现基础身份/权限边界和授权视图 |
| 集群化水平扩展 | 否 | 后续迭代考虑 |

### 3.6 成功标准

| 类型 | 可验收标准 |
| --- | --- |
| 功能成功 | 助教能够完成一次从提交查看、评测触发、结果查看到榜单发布的演示闭环 |
| 公平性成功 | 同一榜单批次使用固定测试集、评分模块、后端参数和提交截止策略 |
| 可复现成功 | 系统保存关键运行参数、后端版本、评测配置和指标结果，支持后续审计 |
| 安全成功 | 学生只能查看自己的授权结果和公开榜单，不能访问他人模型权重、隐藏测试集、内部日志 |
| 可解释成功 | 排行榜不仅给出综合分，还能说明主要指标和异常状态 |
| TSP 成功 | 方案中能清楚回答目标、角色、计划、质量、风险和团队承诺从何而来 |

### 4.3 角色组合理由

本团队采用兼任方式覆盖 TSP 典型角色：

1. 项目组长兼过程经理，便于把会议组织、过程记录、文档版本和团队承诺统一维护。
2. 开发经理兼支持经理，便于把技术路线、工具环境、复用策略和配置管理结合起来。
3. 计划经理独立设置，保证两周计划、估算依据、负载平衡和进度跟踪有明确责任人。
4. 质量经理独立设置，保证质量计划不是最后测试兜底，而是通过评审、过程控制和指标审计前移。
5. 所有成员仍承担开发或交付任务，管理角色只代表过程责任，不代表其他成员不参与讨论或交付。

### 4.4 团队目标

团队目标与课程和专家评审目标保持一致：

1. 在 2 周启动周期内形成可演示、可解释、可复现的 MVP 方案和交付计划。
2. 用 TSP Launch 方式完成目标澄清、角色分配、开发策略、整体计划、质量计划、计划平衡和风险评估。
3. 不追求一开始覆盖所有高级功能，而是优先保证助教评测闭环可信。
4. 所有关键结论都要在阶段一方案中沉淀，便于阶段二直接引用。
5. 团队承诺来自共同讨论和角色分工，而不是单人决定。
