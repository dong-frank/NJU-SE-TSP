## 3. 前三次会议结论对本次计划的约束

### 3.1 第一次会议输入：目标与范围

第一次会议明确，FIM评测平台不是普通OJ，也不是单纯的评测脚本，而是面向大语言模型代码补全能力评测的服务化平台。平台需要把学生模型提交、模型目录校验、统一推理、指标计算、报告生成和榜单发布组织成可追踪、可复现、可比较的流程。

因此，本次计划必须满足以下约束：

- 计划必须覆盖助教评测闭环，不能只规划一个榜单网页。
- 计划必须体现公平性、可复现性和安全过滤。
- 计划必须明确MVP范围和非MVP范围，防止两周目标失控。

### 3.2 第二次会议输入：角色与责任

第二次会议确认4人团队角色如下：

| 成员 | TSP管理角色 | 本次会议关注点 |
| --- | --- | --- |
| 饶东申 | 项目组长、过程经理 | 计划是否能支撑策划书和阶段二答辩，会议产物是否可追踪 |
| 张珂铫 | 质量经理 | 计划中是否安排评审、质量前移、指标公平性和榜单安全检查 |
| 刘汇鑫 | 计划经理 | 组织章节分工、排期、资源判断，并为第六次会议准备WBS和估算输入 |
| 邵勃源 | 开发经理、支持经理 | 技术路线、复用边界、工具环境和支撑条件是否可行 |

团队再次确认，所有成员首先都是开发人员，同时兼任TSP管理角色。角色负责人负责组织相关讨论和产出，但计划仍需由全体共同承诺。

### 3.3 第三次会议输入：开发策略与过程

第三次会议由开发经理兼支持经理邵勃源主持，已形成开发策略和过程定义。刘汇鑫在本次会议中将第三次会议结论整理为本次计划输入，并请邵勃源确认。

## 6. 14天平台MVP详细计划

### 6.1 总体开发策略与范围

会议主要讨论了FIM评测平台MVP的开发策略与两周内的详细计划，明确了各阶段需开发和复用的功能范围。

总体策略是在两周内打通助教评测的完整闭环，实现模型提交、校验、评测、报告查看及榜单发布的核心功能。复用部分包括现有MindSpeed评测脚本、指标计算方法及公开榜单生成能力；自研部分包括服务化封装、任务管理、并发排队、指标可配置、安全过滤及数据存储等模块。

集群化扩展、完整多后端资源调度、复杂健壮性处理、完整学生门户和复杂artifact下载等非核心功能可延后处理。

### 6.2 FR-01到FR-11的MVP覆盖

会议确认，FR-01到FR-11都需要进入策划书，但承诺层级不同：核心闭环作为MVP强承诺，复杂自动化和完整工程化能力作为降级实现或后续迭代。

| FR编号 | 功能模块 | 14天MVP交付物 | 范围说明 |
| --- | --- | --- | --- |
| FR-01 | 课程提交拉取 | 提交同步/导入模块 | 支持按课程、作业、名单和日期策略导入或同步提交；OBS/STS全自动链路保留接口边界 |
| FR-02 | 模型根目录校验 | 模型根目录校验器 | 检查配置、tokenizer、权重、推理入口等必要文件 |
| FR-03 | 多测试集评测 | 测试集配置与任务参数 | 支持配置化选择样本范围和任务口径，完整多测试集管理后续扩展 |
| FR-04 | 多推理后端 | 后端统一契约与主链路适配 | 优先支持MindSpeed主链路或mock/raw演示链路，保留多后端扩展接口 |
| FR-05 | 结果恢复与指标计算 | generation JSONL恢复与核心指标计算 | 计算EM、ES、TLE通过率、非复读率等核心指标 |
| FR-06 | 内部聚合报告 | 内部报告生成 | 输出内部报告，支持助教查看失败归因和基线对比 |
| FR-07 | 公开榜单 | 安全过滤后的公开榜单 | 生成公开榜单，不泄露模型路径、逐样本内容和内部日志 |
| FR-08 | 状态管理 | 评测任务状态机 | 标记pending、running、ok、absent、tokenizer_error、missing_raw_result等状态 |
| FR-09 | 审计与复现 | 运行元数据与配置快照 | 保存数据集、评分模块、后端版本、运行参数、样本数和日志摘要 |
| FR-10 | 服务化接口规划 | Web控制台/API骨架 | 实现同步、评测创建、进度查询、榜单查询等核心接口骨架 |
| FR-11 | 实时日志可视化 | 阶段进度与日志视图 | 助教查看阶段、日志、进度和失败原因；学生只看授权摘要 |

### 6.3 非MVP强承诺范围

学生基础入口属于MVP范围，包括提交模型、查看本人评测分数、查看提交状态、查看必要结果摘要和查看公开榜单。以下能力不作为14天MVP强承诺，但保留为后续迭代方向：

- 多推理后端的完整自动调度和资源隔离。
- 完整学生鉴权注册流程、个人中心、多作业历史筛选和详细报告下载。
- 复杂artifact下载接口。
- AI助教对每份提交生成深入改进建议。
- 多课程、多作业的完整管理体系。
- 榜单回滚服务的完整工程化。
- 集群化部署、水平扩展和全量故障演练。

## 7. 资源判断与计划风险

会议对资源进行了集中讨论。刘汇鑫提出：14天名义资源为56人日，但考虑课程压力、沟通成本、评审活动和不确定性，实际有效投入不能按满负荷计算。

原始计划将FR-04主推理链路或mock/raw演示链路接入只放在Day 6，团队讨论后认为这一安排过于靠后，也没有体现它的复杂度。调整后，模型校验、测试集和样本范围配置等相对简单内容前置到Day 3；FR-04集中安排在Day 5-Day 6完成后端契约和调用适配，使后续Day 7-Day 8可以继续接结果恢复、指标计算和主流程联调。

团队确认，计划成立需要满足四个条件：

1. 不再扩大MVP范围。
2. 复用已有MindSpeed评测脚本、指标计算方法和公开榜单生成能力。
3. 保留mock/raw结果作为演示兜底。
4. 将模型校验、任务配置等相对简单工作前置，给FR-04主链路在Day 5-Day 6预留连续接入时间。

如果后续发现资源不足，团队同意采用如下降级顺序：

1. 保留助教闭环主链路。
2. 保留指标计算、报告和公开榜单。
3. 保留审计元数据和安全过滤的最小集合。
4. 降级实时日志细节、完整学生门户、多后端调度、复杂artifact下载、集群化扩展和完整健壮性处理。

该降级顺序将作为第六次计划平衡会议和第七次风险评估会议的重要输入。

## 8. 已形成决策

1. 第四次会议采用计划经理提出草案、团队共同讨论确认的方式制定整体计划。
2. 项目计划区分两个时间层次：3天完成《项目策划方案》，14天上线平台MVP。
3. 第一阶段正式提交物为《项目策划方案》，会议材料作为策划方案证据和素材。
4. 策划书组织上先呈现章节分工、14天详细计划、资源判断和计划风险；WBS、估算和个人任务平衡留到第六次会议详细展开。
5. 14天MVP优先覆盖助教评测闭环，同时保留学生基础入口。
6. 14天MVP复用现有MindSpeed评测脚本、指标计算方法和公开榜单生成能力。
7. 自研重点包括服务化封装、任务管理、并发排队、指标可配置、安全过滤和数据存储。
8. 原Day 5的模型校验、测试集配置等相对简单内容前置到Day 3-Day 4。
