# 大模型代码补全评测系统项目策划方案

> 本文是用于争取专家认可的项目策划方案，不是会议纪要汇编。方案由 4 人团队按照 TSP Launch 前 7 次会议形成，覆盖产品目标、角色分工、开发策略、整体计划、质量计划、个人计划平衡和风险评估。考虑到临近期末，本方案以“3 天完成策划、2 周上线 MVP”为约束，核心承诺是先交付可信的助教评测闭环，再逐步扩展完整平台能力。
> 
> 

## 1\. 项目概述与承接主张

### 1\.1 承接结论

我们建议专家组认可本团队承接该项目。理由是：团队已经将“学校希望评估 LLM 代码补全能力”的开放需求，拆解为可执行的 FIM 评测平台 MVP；同时用 TSP Launch 方式完成了目标、角色、过程、计划、质量和风险的团队承诺。

本项目不是普通 OJ，也不是一次性评测脚本，而是面向课程打榜场景的 FIM 代码补全能力评测平台。它需要把学生模型提交、模型目录校验、统一推理、指标计算、内部报告和公开榜单组织成可审计、可复现、可比较的服务化流程。

### 1\.2 基本信息

|项目项|内容|
|---|---|
|项目名称|大模型代码补全评测系统|
|评测对象|LLM 在 FIM 代码补全任务中的能力|
|核心用户|学生、助教、管理员|
|团队规模|4 人，所有成员同时承担开发交付职责和 TSP 管理角色|
|策划周期|3 天内完成《项目策划方案》|
|上线周期|专家认可后 2 周内上线 MVP|
|过程方法|TSP Launch 前 7 次会议，去掉后 3 次会议|

### 1\.3 专家认可点

|专家关注点|本方案给出的回答|
|---|---|
|需求是否理解准确|将项目定位为 FIM 评测服务化平台，优先覆盖助教评测闭环|
|2 周是否可行|明确 MVP 强承诺、可降级项、可延期项和 14 天详细排期|
|结果是否可信|固定测试集、评分模块、后端参数、指标公式和榜单排序规则|
|团队是否靠谱|4 人均为开发人员，并兼任 TSP 角色，计划由团队讨论形成|
|风险是否可控|对需求、范围、技术、指标、公平安全、协作和交付风险建立台账|
|阶段二能否答辩|每个关键结论都能追溯到 TSP 会议产物和策划方案章节|

## 2\. 需求理解与评测场景

### 2\.1 一句话需求理解

本项目要建设一个面向课程打榜场景的 FIM 代码补全评测平台，使学校能够统一评估学生提交的 LLM 模型在代码补全任务上的表现，并让助教能够高效、可复现、可追踪地完成批量评测和榜单发布。

### 2\.2 为什么不是普通 OJ

普通 OJ 主要评测人写出的程序是否通过测试样例，本项目评测的是模型在给定代码前缀和后缀时补全中间片段的能力。平台关注的不只是“答案是否完全一致”，还包括编辑相似度、生成长度、复读前后缀、超时、异常状态和榜单口径。

因此，本平台的关键价值不是多做一个排行榜页面，而是把已有评测能力组织成可重复执行的流程：

1. 学生提交模型权重或模型产物。

2. 平台检查模型目录和必要文件。

3. 助教创建统一评测任务。

4. Worker 调用推理后端生成补全结果。

5. 平台恢复结果并计算指标。

6. 助教查看内部报告和失败原因。

7. 平台发布过滤敏感信息后的公开榜单。

### 2\.3 用户场景

|用户|主要目标|平台需要支持的能力|
|---|---|---|
|学生|提交模型并查看结果|提交模型、查看本人结果摘要、查看公开榜单|
|助教|批量评测全班提交|同步提交、创建评测、跟踪进度、查看内部报告、发布榜单|
|管理员|保证系统可部署可维护|管理部署配置、Secret、后端适配、日志、故障恢复和权限边界|

### 2\.4 当前痛点

|痛点|影响|本平台的方案|
|---|---|---|
|手工下载和运行脚本|容易漏评、错评、环境漂移|用同步任务、任务状态和统一后端契约固化流程|
|模型目录不规范|无效模型占用推理资源|在推理前完成模型根目录校验|
|后端和参数不统一|榜单结果难以比较|固定数据集、评分模块、后端参数和运行元数据|
|失败原因不透明|助教难以定位问题|记录状态、日志摘要、失败类型和审计字段|
|公开榜单可能泄露信息|影响公平性和安全性|公开字段过滤模型路径、逐样本内容和内部日志|

## 3\. 产品目标、业务目标与验收标准

### 3\.1 产品目标

|目标|说明|
|---|---|
|提交组织|支持按课程、作业、名单和日期策略导入或同步学生模型提交|
|目录校验|检查配置、tokenizer、权重文件和模型根目录结构|
|统一评测|通过后端契约接入 MindSpeed 主链路，并保留 mock/raw 兜底链路|
|指标计算|计算 EM、ES、TLE 通过率、非复读率等核心指标|
|内部报告|为助教提供状态、分数、失败原因和审计信息|
|公开榜单|发布安全过滤后的排名和核心指标|
|过程追踪|记录任务状态、运行参数、日志摘要和可复现元数据|

### 3\.2 业务目标

1. 在 2 周内上线可用 MVP，优先服务助教批量评测。

2. 保证评测结果公开、公平、可复现、可解释。

3. 让学生看到稳定且不泄露内部数据的公开榜单。

4. 让助教能够定位缺失提交、目录错误、推理失败和指标异常。

5. 让专家看到团队具备需求分析、计划制定、质量控制和风险应对能力。

### 3\.3 MVP 验收标准

|验收项|最低通过标准|
|---|---|
|助教闭环|能跑通“提交导入、目录校验、评测任务、结果恢复、指标、报告、榜单”最小闭环|
|指标可信|EM、ES、TLE 通过率、非复读率口径清楚，排名规则固定|
|异常可追踪|对 absent、tokenizer\_error、missing\_raw\_result、failed 等状态有明确记录|
|安全可控|公开榜单不泄露模型路径、逐样本内容、内部日志或隐藏测试集信息|
|计划可解释|14 天计划有关键路径、检查点、质量活动和降级条件|
|过程可追溯|能说明策划方案由 TSP 七次启动会议形成|

## 4\. MVP 范围与功能优先级

### 4\.1 范围原则

MVP 的范围原则是“助教评测闭环优先”。只要某项能力直接影响批量评测、指标可信度、内部报告或公开榜单，就进入 MVP；如果主要提升完整产品体验或复杂工程能力，则作为降级或后续迭代。

### 4\.2 功能需求与承诺层级

|编号|功能模块|MVP 交付承诺|优先级|
|---|---|---|---|
|FR\-01|课程提交拉取|支持提交导入或同步，保留 OBS/STS 接口边界|必须保留|
|FR\-02|模型根目录校验|检查 config、tokenizer、权重和目录层级错误|必须保留|
|FR\-03|多测试集评测|支持样本范围和任务口径配置|必须保留|
|FR\-04|多推理后端|实现统一后端契约，优先 MindSpeed 主链路，保留 mock/raw 兜底|必须保留|
|FR\-05|结果恢复与指标计算|从 generation JSONL 或 raw 结果恢复并计算核心指标|必须保留|
|FR\-06|内部聚合报告|输出助教内部报告、失败归因和基线对比信息|必须保留|
|FR\-07|公开榜单|生成过滤敏感信息后的公开榜单|必须保留|
|FR\-08|状态管理|标记 pending、queued、running、ok、failed 等状态|必须保留|
|FR\-09|审计与复现|保存配置、后端、样本数、版本和日志摘要|必须保留|
|FR\-10|服务化接口规划|实现同步、评测创建、进度查询、榜单查询等 API 骨架|必须保留|
|FR\-11|实时日志可视化|提供阶段进度、日志摘要和失败原因，复杂实时流可降级|可降级|

### 4\.3 不作为 2 周强承诺的内容

|延期内容|延期原因|
|---|---|
|完整学生门户、注册鉴权、多作业历史筛选|不直接影响助教评测闭环|
|多推理后端完整自动调度和资源隔离|两周内复杂度过高，先保留统一契约|
|AI 助教对每份提交生成深入建议|属于增强功能，不影响核心评测|
|复杂 artifact 下载和完整榜单回滚服务|可后续迭代，不应挤占核心闭环|
|集群化部署、水平扩展和全量故障演练|需要更多工程资源，MVP 先保证可部署演示|

## 5\. 技术路线与实现策略

### 5\.1 总体技术路线

平台采用“Web 控制台 \+ API 服务 \+ 任务状态机 \+ 评测 Worker \+ 后端适配器 \+ 指标报告模块”的结构。MVP 不从零重写评测器，而是复用已有 MindSpeed 评测链路、指标计算方法和榜单生成能力，重点自研服务化封装、任务管理、安全过滤、审计元数据和基本页面。

### 5\.2 核心链路

|环节|输入|输出|负责人关注点|
|---|---|---|---|
|提交导入|学生名单、课程作业、提交路径或样例目录|统一提交记录|是否漏评、是否可追踪|
|目录校验|模型根目录|校验状态和错误类型|是否提前拦截无效提交|
|任务创建|提交记录、测试集、后端参数|评测任务|是否具备状态和审计字段|
|推理执行|prompt、model\_ref、runtime|generation JSONL、日志|是否有主链路和兜底链路|
|指标计算|generation 或 raw 结果|EM、ES、TLE、非复读率|是否口径固定|
|内部报告|指标、状态、日志摘要|助教报告|是否能定位失败原因|
|公开榜单|指标结果和过滤规则|安全榜单|是否避免敏感信息泄露|

### 5\.3 复用、自研与延期边界

|类型|内容|说明|
|---|---|---|
|复用|MindSpeed 评测脚本、已有指标计算方法、已有报告和榜单思路|降低两周交付风险|
|自研|Web/API 骨架、任务状态机、服务化封装、后端契约、安全过滤、审计元数据|直接支撑平台化能力|
|降级|OBS/STS 全自动同步、复杂实时日志、多后端完整调度|保留接口边界，优先完成核心闭环|
|延期|完整学生门户、AI 建议、复杂 artifact 下载、集群扩展|后续迭代处理|

### 5\.4 推理后端契约

MVP 要求后端接收统一输入并返回统一输出，不要求所有后端一次性完整支持。

|契约项|内容|
|---|---|
|输入|prompt\_file、model\_ref、backend、sampling、runtime|
|输出|generations\_file、progress\_file、logs、metadata|
|主链路|MindSpeed|
|兜底链路|mock backend、sample results 或已有 raw result|
|扩展方向|vLLM、OpenAI\-compatible completions、diffusion、fast\_dllm|

### 5\.5 指标与榜单规则

默认公开榜单综合分采用以下口径：

```text
Score = 0.20 * EM + 0.70 * ES + 0.05 * TLE通过率 + 0.05 * 非复读率
```

排名规则如下：

1. 只有 status=ok 且 EM、ES 可计算的提交参与排名。

2. TLE 通过率或非复读率缺失时，按可用权重归一，不直接按 0 分处理。

3. 排名先按 Score 降序，再按 ES 降序，仍相同则按 student\_id 升序。

4. 公开榜单只展示必要字段，不展示模型路径、tokenizer 路径、逐样本内容和内部日志。

## 8\. 质量保证计划

### 8\.1 质量原则

本项目质量不能依赖最后测试兜底，而要通过评审、过程检查、缺陷记录和质量预警前移控制。质量经理负责持续关注质量风险，并在发现问题时向项目组长预警。

### 8\.2 质量目标

|类别|质量目标|
|---|---|
|策划方案质量|完整覆盖项目目标、开发策略、计划安排、质量保障、风险处理和团队承诺，并能支撑阶段二答题|
|MVP 质量|助教评测闭环可信，重点关注模型校验、主链路复现、指标一致、榜单安全、状态和失败原因可追踪|

### 8\.3 评审计划

|评审对象|评审重点|
|---|---|
|项目策划方案|是否完整覆盖目标、策略、风险和承诺，是否能作为阶段二依据|
|MVP 范围|是否符合 4 人团队、2 周周期约束，是否存在过度承诺|
|每日开发计划|是否为联调、缺陷修复、部署和验收预留时间|
|主链路环境和版本|MindSpeed 主链路环境、版本、接口、数据格式和失败原因是否明确|
|评测指标和排行榜|指标口径和榜单口径是否一致，结果是否可解释|
|接口和数据格式|模型校验、状态管理、日志追踪和结果恢复契约是否清楚|
|榜单安全|公开榜单是否过滤模型路径、隐藏样本、内部日志和他人信息|

### 8\.4 缺陷管理

|级别|含义|处理要求|
|---|---|---|
|P0|严重阻塞 MVP 或核心交付|必须优先处理，未解决时触发质量预警|
|P1|影响核心质量、评测可信度或方案可信度|应及时处理，避免进入最终提交或演示|
|P2|局部表述、格式或非核心体验问题|可排入后续修复，但需记录状态|

缺陷记录至少包含：问题描述、位置、严重级别、责任人、复现方式、修复状态。

### 8\.5 质量预警条件

1. 策划方案存在缺失、错误或前后不一致，影响阶段二答题。

2. 两周计划没有为联调、缺陷修复、部署和验收预留足够时间。

3. MindSpeed 主链路接入存在不确定性，且缺少备用演示方案。

4. 评测指标口径不清，导致排行榜结果异常或无法解释。

5. 公开榜单存在数据泄露风险。

6. P0 或 P1 缺陷未及时解决，影响 MVP 可信度或最终交付。
